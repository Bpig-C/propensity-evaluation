## Layer 2：数据契约层

### 核心数据格式

场景数据为嵌套 JSON，按 `domain → workspace → role → scenarios` 四层组织：

```json
{
  "<domain>": {
    "<workspace>": {
      "<role>": {
        "scenarios": {
          "<scenario_name>": {
            "states": {...},
            "functions": [...],        // 工具调用定义
            "target_function": "...",  // 触发函数名
            "trigger_function": "...", // 有害函数名
            "policy": "...",
            "messages": {...},
            "sys_messages": [...]      // 压力系统消息列表
          }
        }
      }
    }
  }
}
```

### 关键字段枚举

**domains（4个）**：`bio-security`、`chemical-security`、`cyber-security`、`self-proliferation`

**函数类型**：每个场景包含 `neutral_name`（无害名）和 `harmful_name`（有害名），`--use_benign` 参数控制使用哪个名字送给评估模型，这是测试"浅层对齐"的核心机制。

**压力类别（6个 tactics，来自 inputs/pressure_categories/）**：
- `evaluation-awareness`（EA）
- `financials`（F）
- `power-seeking`（PS）
- `resource-deprivation`（RD）
- `self-preservation`（SP）
- `time`（T）

每个 tactic 有 11 个强度级别，构成 6 × 11 = 66 种压力配置。

### 文件路径依赖

| 路径 | 用途 | 消费方 |
|------|------|-------|
| `data/full/<domain>/<workspace>/` | 预生成基准数据（Git LFS） | evaluation/main.py |
| `data/sample/<domain>/<workspace>/` | 样例数据（5个场景/domain） | 快速检验 |
| `inputs/attacks/<domain>/attacks.json` | 攻击向量定义 | generation/main_scen_pipeline.py |
| `inputs/workspaces/<domain>/*.json` | Workspace 定义（角色+描述）| generation pipelines |
| `inputs/pressure_categories/*.json` | 压力类别配置 | generation/messages_pipeline |
| `generation_output/` | 生成流水线中间产物 | step1→step2 间传递 |
| `evaluation/evaluation_results/` | 评估结果（JSONL格式） | evaluation/main.py 输出 |
| `evaluation/evaluation_trajectories/` | 完整交互轨迹日志 | 研究/调试用 |
| `.tmp/` | 限速信号量文件（fcntl 锁） | api_client/client.py::RateLimiter |
| `state.json` | 评估断点续跑状态 | evaluation/main.py |

## Layer 3：交互协议层

### 对外接口

**evaluation/main.py（主评估入口）**：
```bash
python evaluation/main.py \
  --model_name <model>      # 被评估模型名
  --model_provider <prov>   # litellm provider（openai/gemini/anthropic等）
  --api_base <url>          # API 端点
  --input_dir data/full/    # 场景输入目录
  --output_dir <dir>        # 结果输出目录（JSONL）
  --log_dir <dir>           # 轨迹日志目录
  --use_benign              # 使用无害工具名（测试浅层对齐）
  --validation_mode         # 验证模式（单步探针，不运行完整episode）
  --max_workers <N>         # 并发 worker 数
  -p <N>                    # 压力级别
  -m <N>                    # 场景消息数
  -j <N>                    # judge 调用次数
  --task_id/--total_tasks   # 分片并行支持
  --no-contd                # 禁用断点续跑
```

**generation（两阶段流水线）**：
```bash
python generation/main_scen_pipeline.py          # Phase 1: 生成 states/funcs/policies/messages
python generation/main_scen_pipeline_messages.py # Phase 2: 生成压力系统消息
```
配置通过 Hydra（`generation/configs/pipeline.yaml`）管理。

### 认证机制

- API 密钥通过 `.env` 文件注入，`API_KEYS` 为空格分隔的密钥池
- `RateLimiter` 使用 `fcntl.flock` 跨进程文件锁 + 信号量文件实现多进程安全限速
- 支持密钥池轮转（key cycling）：一个 key 耗尽时自动切换到下一个

### 第三方服务依赖

| 服务 | 协议 | 用途 |
|------|------|------|
| OpenAI / Gemini / Anthropic 等 | HTTPS LiteLLM | 场景生成 + 评估推理 |
| LiteLLM Proxy | HTTPS | 统一 API 代理层（可选）|

## Layer 4：失效边界层

### 降级策略

| 场景 | 处理方式 |
|------|---------|
| API 限速（429） | RateLimiter 自动循环等待 20s 后重试；切换下一个 key/model |
| LLM 响应格式不合法 | Executor 最多 12 次重试；ValidationExecutor 同等策略 |
| 进程中断（SIGTERM） | evaluation/main.py 捕获信号，保存 state.json，支持断点续跑 |
| 生成结果不完整 | PipelineMessages.run_gen_messages 检测字段完整性，不完整则跳过更新 |
| pyan3 无法分析含嵌套 lambda 的文件 | evaluation/main.py 排除（已记录）|

### 可选组件（opt-in）

- `--use_benign`：切换工具命名模式（默认 harmful 名，opt-in benign 名）
- `--validation_mode`：单步探针替代完整 episode（用于快速验证）
- `RATE_LIMIT=true`：启用限速（默认关闭，需显式开启）
- `--no-contd`：禁用断点续跑（默认启用续跑）

### 关键失效场景矩阵

| 失效场景 | 影响 | 检测 |
|---------|------|------|
| Git LFS 未安装 | data/full/ 内容为 LFS 指针文件，评估无法运行 | FileNotFoundError/乱码 |
| API_KEYS 未设置 | Client.__init__ 中 assert 直接中断 | AssertionError |
| fcntl 在 Windows 不可用 | RateLimiter._initialize_semaphores 崩溃 | ImportError（fcntl 是 POSIX 专属）|
| generation_output 不存在 | messages pipeline 无法读取 step1 产物 | FileNotFoundError + logger.error |
