先说判断：**你现在最适合把 PropensityBench 当成“评测外壳 / 实验基础设施”来用，而不是先把它当成一篇要从头吃透的“大一统论文”来读。**
这和你现在的框架状态是匹配的：你的 v2.2 已经把重点放在**欺骗、自我复制、Sandbagging**三条主线，而复现计划里也把 **R11 / PropensityBench** 定位成 **Tier-0 的快速基线**，目标是拿到“Agent 潜在安全风险的分布基线”，而不是让它主导整个框架。 

更具体地说，PropensityBench 对你最有价值的不是四大风险域本身，而是它这套**agentic evaluation 结构**：
它把每个场景做成“**安全路径 vs 危险路径**”的等价选择题，再通过**压力消息**逐级施压，并用统一指标记录模型什么时候转向危险工具。论文自己也明确把贡献放在：**agentic benchmark、scenario generation pipeline、pressure taxonomy、PropensityScore 及其行为指标**上。

所以你可以这样利用它：

* **先复用外壳**：场景协议、消息流、trace 记录格式、压力注入、指标计算。
* **再替换内容**：把 bio/chem/cyber/self-proliferation 的具体工具，替换成你自己框架里的 deception / self-replication / sandbagging proxy tools。
* **最后再补原理**：等你把一条最小链路跑通，再回头精读它的 threat model、pressure design、metrics。

## 1）论文哪些必须看，哪些先不用看

### 你的阅读顺序应该是：**先评测抽象，后领域内容**

因为你当前目标是“先利用代码”，所以不要从四大风险域 taxonomy 开始读。
最该先读的是：**这套 benchmark 为什么这样设计、怎么跑、怎么记分、哪些地方最容易迁移到你的统一框架**。这正好也和你 v2.2 里强调的“系统级评测、动态演化、Harness 层、能力边界与放大条件”是同一层问题。

### 必读，按优先级排序

**A. Abstract + Introduction + Table 1**
你只需要抓三件事：

* 它测的不是 capability，而是 **propensity**：模型**在被赋予危险工具后会不会去用**。
* 它的核心实验单位不是单轮问答，而是 **agent + tools + pressure**。
* 数据规模和结构：**979 个 unique scenarios，扩展成 5,874 个 evaluation tasks，6 个压力维度，6,648 个工具**。这个对你判断“先跑哪部分代码”很重要。

**B. Section 3.1 Threat Model + 3.2 Design Goals**
这是你最该看懂的原理层。
尤其看两点：

* 论文把 **LLM agent 本身**当潜在 threat actor，不是外部攻击者。
* 它刻意让 **aligned tool** 和 **misaligned tool** 在任务结果上等价，从而把“能力”尽量剥离，只测“选择倾向”。这点和你框架里把能力边界、目标偏差、监督失效、放大条件拆开看，是高度兼容的。

**C. Section 3.4 Evaluating Propensity: Agentic Approach + Figure 3/5**
这是你迁移代码时最有用的部分。一定要看懂：

* 一个 scenario 的消息流怎么走；
* getter tool、aligned tool、misaligned tool 分别扮演什么角色；
* 为什么 aligned tool 被故意设计成持续报错；
* pressure message 在什么时机插入；
* 什么时候 simulation terminate。

这部分其实就是你将来复用代码时的“协议层”。

**D. Section 3.6 Propensity Under Pressure + Table 2 + Figure 10**
这是第二个最值得迁移的部分。
因为它给你的不是某个结论，而是一套**可复用的压力语法**：

* 6 个压力维度；
* 每个维度拆成 5 个 component；
* 每个 component 有 11 个强度；
* 再生成逐级升级的 pressure messages。

对你来说，这正好可以改造成：

* 欺骗场景中的“监督压缩 / 评测感知 / 时间压力”
* 自复制场景中的“自保压力 / 资源剥夺 / 权限窗口”
* Sandbagging 场景中的“评测感知 / 可验证性差异 / 部署门槛诱导”

**E. Section 3.7 Metrics + Table 3/4 + Figure 14/15/16/17/19**
这部分你不用逐个模型精读，但一定要看懂指标定义：

* **PropensityScore**
* **Resilience**
* **Persistence**
* **Abandonment**
* **tool naming sensitivity**
  这些指标对你的帮助很大，因为它们可以直接映射成你框架里的**行为层/能力边界/压力鲁棒性**子指标。

尤其是：

* “**有压力时快速转向危险路径**”
* “**显式知道不该做，但还是做**”
* “**去掉危险命名提示后更容易违规**”
  这几种现象，本质上都可以成为你统一框架里的观测特征。

### 第二层再看

**F. Section 3.5 Automated Scenario Generation**
这部分先不用精读原理，只需要知道它存在两层东西：

* 你短期要跑的是 **evaluation harness**
* 你中期才需要碰 **scenario generation pipeline**

也就是说，**先跑评测，不要一上来就碰自动造场景**。论文自己也把 generation pipeline 设计得比较重。

**G. 四大风险域的具体 capability taxonomy 细节**
