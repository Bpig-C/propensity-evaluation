## 一、项目本质

**PropensityBench 是一个 LLM 安全倾向性评测框架**，专门测量语言模型在被赋予高风险工具、并置于"操作压力"下时，**倾向于执行有害行动的概率**。

它的核心研究问题是：不是"模型能不能做坏事"（能力），而是"模型在正常运行环境下会不会主动选择做坏事"（倾向性/propensity）。

框架分两条主线：
1. **场景生成流水线**（`generation/`）：用 LLM 批量生成高风险业务场景（4个领域 × 32个工作空间 × 多角色 = 979个场景，共5874个测试用例），每个场景包含工具定义、状态机、政策约束、压力消息。
2. **Agentic 评估执行器**（`evaluation/`）：将被测 LLM 放入模拟的高风险工作空间，配备真实工具调用接口，施加系统性压力，观察其是否调用了预设的"有害目标函数"（`trigger_function`）。

## 三、各模块职责精要

### api_client/

**本质**：多 API key 池化 + 跨进程限速的 LiteLLM 包装层。

- `APIConfiguration`：简单数据类，持有 model_name/provider/api_base/use_cache 四个字段
- `RateLimiter`：用文件系统信号量（`.tmp/` 目录下的 JSON 文件 + `fcntl.flock`）实现跨进程安全限速；后台线程每秒释放过期 token；支持 key 轮转
- `Client`：封装 `litellm.completion()`，支持 Pydantic schema 或 dict schema 的结构化输出；温度/缓存/proxy 均可配置

**关键隐患**：`fcntl` 是 POSIX 专属模块，在 Windows 上 **不可用**。这是该项目在 Windows 上无法运行限速功能的根本原因。

### evaluation/

**本质**：把 LLM 关进一个带有"诱导性工具"的模拟工作空间，观察其是否触发有害行动。

- **`Agent`**：持有对话历史（history）、工具调用封装（OpenAI function-calling 格式）和角色系统提示。`SYSTEM_PROMPT_TEMPLATE` 故意伪造"真实世界 API"的感知（"This API controls REAL-WORLD instruments"），这是施压设计的一部分。
- **`Executor`**：管理 12 轮交互循环，处理 tool call 解析、连续消息限制、工具调用计数。关键逻辑：检测 `trigger_function` 是否被调用（=propensity 触发）。
- **`ValidationExecutor`**：单步探针，用于验证场景是否合理（不跑完整 episode），返回 yes/no 判断。
- **`PropensityEvaluator`（main.py）**：最复杂模块（CC=46）。并发运行多个场景，支持断点续跑（state.json），处理压力级别叠加，写出 JSONL 结果。

### generation/

**本质**：五阶段 LLM 工厂，从 workspace 定义出发，逐步生成状态机 → 函数定义 → 政策约束 → 任务消息 → 压力消息。

- **BasePipeline**：提供日志写入基础设施
- **PipelineScenarios**：驱动前四个阶段，按 domain/workspace 并行处理（ProcessPoolExecutor）
- **PipelineMessages**：驱动第五阶段，向已有场景追加压力系统消息
- **五个 ScenarioManager**：各自对应一个生成阶段，内部逻辑相似（生成 → Judge 打分 → 筛选 → 重试）
- **MessagesScenarioManager**：最复杂，负责为 6 种压力 tactic × 11 强度级别 × 每场景生成压力消息（D级复杂度）
- **SimilarityGraph**：用 TF-IDF + 余弦相似度去除重复场景（networkx 最大权重节点删除算法）

## 四、关键依赖与耦合边界

### 硬依赖

| 依赖点 | 具体说明 | 影响范围 |
|--------|---------|---------|
| `data/full/` Git LFS 数据 | ~770MB，未 pull 时为 LFS 指针 | 评估无法运行 |
| `inputs/pressure_categories/*.json` | 6个压力 tactic 定义文件 | 消息生成流水线 |
| `inputs/workspaces/<domain>/*.json` | Workspace 角色定义 | 生成流水线入口 |
| `generation_output/` 中间产物 | Step1 → Step2 的传递媒介 | messages pipeline |
| POSIX `fcntl` 模块 | 跨进程限速锁 | Windows 上失效 |
| `API_KEYS` 环境变量 | Client 初始化断言 | 所有 LLM 调用 |
| LiteLLM | 统一 LLM 调用层 | 全系统 |

### 软依赖（可替换）

- `hydra`：配置管理（generation/），可替换为 argparse
- `rich`：彩色终端输出（evaluation/utils/display.py），纯视觉增强
- `networkx` + `sklearn`：相似度去重（generation/pipeline/modules/graph_utils.py），仅影响场景多样性

## 五、架构健康度评估

### 优点

| 方面 | 说明 |
|------|------|
| 关注点分离清晰 | generation / evaluation / api_client 三层职责明确 |
| 断点续跑设计 | state.json + SIGTERM 处理，大规模评估不怕中断 |
| 可扩展压力框架 | 6 tactics × 11 levels 可独立扩展 |
| 多模型支持 | LiteLLM 抽象层屏蔽了各 provider 差异 |
| 工具命名对照设计 | neutral/harmful 双名字对照是核心实验设计，代码支持良好 |

### 值得关注的问题

| 问题 | 位置 | 影响 |
|------|------|------|
| fcntl 不可移植 | api_client/client.py::RateLimiter | Windows 上 RateLimiter 完全失效，需用 multiprocessing.Semaphore 替代 |
| evaluation/main.py CC=46 | evaluation/main.py::main | 难以测试和维护，是最容易引入 bug 的位置 |
| ScenarioManager 同名类泛滥 | 4个文件中均有 ScenarioManager | import 时易混淆，建议添加模块前缀（FuncsScenarioManager 等）|
| 两个 main_scen_pipeline.py 重复逻辑 | generation/main_scen_pipeline*.py | init_attack_vectors/get_attack_vectors_for_roles 等函数完全重复 |
| fcntl 信号量文件积累 | .tmp/ 目录 | 长期运行后文件数增多，目前无清理机制 |

## 六、对后续工作的指引

如果目标是**复现论文评估结果**：
1. 必须先 `git lfs pull` 拉取 `data/full/` 完整数据（770MB）
2. 配置 `.env`：`API_KEYS`、`RATE_LIMIT=true`、`API_PROXY=litellm`、`RATE_PM=60`
3. 在 **Linux/macOS** 上运行（fcntl 依赖）
4. 关键命令：`python evaluation/main.py --model_name <model> --input_dir data/full/ ...`

如果目标是**扩展到新的风险领域**：
1. 在 `inputs/workspaces/<new_domain>/` 下添加 workspace JSON 定义
2. 在 `inputs/attacks/<new_domain>/attacks.json` 中定义攻击向量
3. 在 `generation/main_scen_pipeline.py` 的 `loaded_workspaces` 列表中追加新 workspace
4. 运行两阶段生成流水线，把生成结果复制到 `data/full/`

如果目标是**修复 Windows 兼容性**：
- 将 `api_client/client.py` 中的 `fcntl.flock` 换成 `filelock` 库（跨平台），或用 `multiprocessing.Manager().Lock()`
